## 1. 어떤 AI를, 어떤 작업에 사용했는가

* **ChatGPT**

  * 프로젝트 방향을 정리하거나 발표 흐름을 구성할 때 사용하였다.
  * 사용자 인터뷰 질문 초안을 빠르게 만드는 데 활용하였다.

* **Gemini**

  * 사용자 시나리오나 UX 흐름을 여러 방향으로 실험해볼 때 활용하였다.
  * 식단 분석 및 개인화 피드백 기능이 실제 서비스로 이어질 수 있는지 조사하는 과정에서 참고하였다.

* **Claude**

  * 시스템 구조를 어떻게 나누면 좋을지, 기능 간 연결을 어떤 방식으로 설계할 수 있을지 아이디어를 탐색하는 데 사용하였다.

* **Figma Make**

  * 초기 와이어프레임과 화면 레이아웃 초안을 빠르게 만들어보는 용도로 활용하였다.

* **Claude Code**

  * Flutter UI를 구현하는 과정에서 레이아웃 예시나 반복되는 UI 코드를 참고하는 용도로 사용하였다.

## 2. AI 제안을 수정하거나 기각한 사례

### 사례 1

초기에는 AI가 일반적인 운동 기록·칼로리 계산 중심의 헬스케어 앱 구조를 많이 제안하였다.

하지만 팀은 단순 피트니스 앱보다는,
“불규칙한 생활을 하는 2030 사용자가 스스로 건강 위험을 인지하고 관리하게 만드는 흐름”이 더 중요하다고 판단하였다.

그래서 운동 기록 중심 화면 대신,
식단·생활 패턴·건강 요약을 먼저 보여주는 방향으로 UX 흐름을 다시 구성하였다.

### 사례 2

AI는 챗봇, 자동 추천, 분석 기능 등을 한 번에 많이 넣는 방향을 자주 제안하였다.

하지만 실제 캡스톤 기간 안에서 구현 가능한 범위를 고려했을 때,
기능 수를 늘리는 것보다 핵심 사용자 흐름과 화면 완성도를 높이는 것이 더 중요하다고 판단하였다.

결과적으로 현재는 Flutter 기반 UI와 사용자 시나리오 중심의 MVP 형태로 우선 개발하였다.

## 3. AI가 틀렸거나 못 믿었던 순간

### 사례 1

AI가 제안한 화면 구조 중에는 실제 서비스 흐름과 맞지 않는 탭이나 기능이 포함되는 경우가 있었다.

특히 기능 수를 늘리는 방향으로 제안되는 경우가 많았는데,
오히려 사용 흐름이 복잡해진다고 판단하였다.

그래서 팀이 직접 사용자 입장에서 화면 이동 흐름을 다시 정리하며 필요한 기능만 남겼다.

### 사례 2

AI는 아직 구현되지 않은 기능까지 실제 서비스에 이미 포함된 것처럼 설명하는 경우가 있었다.

팀은 발표와 문서에서 구현 완료 기능과 아이디어 단계 기능을 명확히 구분해야 한다고 판단하였다.

그래서 README와 발표 자료에서도 실제 구현 범위를 기준으로 내용을 다시 정리하였다.

## 4. AI 없이 우리가 직접 한 것

* **서비스 방향성과 타겟 사용자 정의**

  * “불규칙한 생활을 하는 2030 위험군”이라는 핵심 사용자 설정은 팀 회의를 통해 직접 정리하였다.

* **발표 시나리오와 UX 흐름 설계**

  * 실제 사용자가 하루 동안 어떻게 서비스를 사용할지를 중심으로 발표 흐름과 사용자 시나리오를 직접 구성하였다.

* **GitHub 협업 및 프로젝트 관리**

  * 브랜치 전략, 역할 분담, PR 및 리뷰 방식은 팀 내부 규칙을 정해 직접 운영하였다.
