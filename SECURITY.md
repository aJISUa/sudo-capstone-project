# Security Policy

On-Care 는 **고혈압·당뇨 위험군 사용자의 식단·운동·혈압·혈당 등 민감 건강 데이터**를 다루는 헬스케어 플랫폼입니다. 본 문서는 보안 취약점 발견 시 보고 절차와 프로젝트의 보안 책임 범위를 정의합니다.

---

## 1. 지원 버전

본 프로젝트는 캡스톤 *스타트* 단계 산출물로, 현재 단일 main 브랜치만 보안 패치 대상으로 지원합니다.

| 버전 | 지원 여부 |
|---|---|
| `main` (latest) | ✅ Supported |
| 그 외 브랜치·태그 | ❌ Unsupported |

---

## 2. 취약점 신고 방법

### 2.1 비공개 채널 (권장)

다음 중 한 가지 방법으로 **비공개**로 보고해 주세요:

- GitHub *Security Advisories* — [Report a vulnerability](../../security/advisories/new) (가능한 경우)
- 이메일: 팀 리더 GitHub 프로필 공개 이메일로 직접 연락

### 2.2 공개 채널 (제한적)

매우 경미하고 데이터·인증과 무관한 이슈만 [Issues](../../issues) 에 `security` 라벨로 등록할 수 있습니다.
**민감 정보 노출 / 인증 우회 / 헬스 데이터 유출 가능성 관련 취약점은 절대로 공개 이슈로 등록하지 마세요.**

### 2.3 신고 시 포함할 내용

- 취약점의 영향 범위 (어떤 데이터 / 어떤 사용자)
- 재현 절차 (가능하면 최소 PoC 포함)
- 영향을 받는 컴포넌트 (예: `frontend/flutter/lib/features/auth/...`)
- 신고자 연락처 (수정 결과 통지용)

---

## 3. 대응 절차 및 일정

| 단계 | 목표 일정 |
|---|---|
| 1차 접수 확인 | 영업일 기준 **2일 이내** |
| 영향도 평가 | 1주 이내 |
| 패치 PR 작성 | 평가 후 2주 이내 (영향도에 따라 조정) |
| 신고자 결과 통지 | 머지 직후 |

---

## 4. 보안 책임 범위

### 4.1 책임 범위 안 (In Scope)

- 인증·세션 관리 결함
- 사용자 입력 검증 부재
- 의존성 패키지의 알려진 CVE
- 백엔드 API 가 노출되는 시점부터의 인증·인가 결함
- 클라이언트 측 민감 데이터 평문 저장
- 배포 워크플로우(`.github/workflows/`) 의 시크릿 노출

### 4.2 책임 범위 밖 (Out of Scope)

- 캡스톤 *스타트* 단계 현재 시점의 mock 인증(소셜 SDK 미연동) — 의도된 한계
- 로컬 DB(drift) 에 저장된 데모용 더미 데이터
- 사용자 디바이스 자체의 OS·브라우저 취약점
- 외부 API (Gemini Vision / 카카오맵 / Firebase 등) 자체 취약점

---

## 5. 헬스케어 데이터 보호 원칙

- 사용자 식단 사진·혈압·혈당·체중 등 **민감 건강 데이터**는 모든 단계에서 다음 원칙을 따릅니다:
  - **최소 수집** — 기능에 필요한 최소 항목만 수집
  - **비식별화** — 외부 전송 시 이름·생년월일 등 PII 마스킹
  - **목적 외 사용 금지** — 분석·연구 목적 활용은 별도 동의 후 IRB 절차 진행
- 그로쓰 단계 진입 시 실제 사용자 데이터 수집을 시작할 경우, **학내 윤리위원회(IRB) 사전 승인** 절차를 거칩니다.

---

## 6. 감사

책임감 있는 신고에 감사드립니다. 신고자의 이름은 (희망 시) 패치 머지 commit·릴리스 노트에 명시됩니다.
