## 2. 면담 질문 및 답변

### Q1. 만성질환 특화 앱임을 어떻게 어필해야 할까요?

진단 영역은 개인정보 문제로 접근하기 어려워 생활 코칭 방향으로 가려고 하는데, 전문성을 어필하기가 애매한 상황

**교수님 답변**

- RAG를 사용하는 만큼 고혈압·당뇨에 특화된 전문 문서를 소스로 쓰는 것 자체가 차별점. 일반 질환 전체를 타겟으로 하는 앱과 달리, 해당 질환 범위 내에서만 답을 찾도록 구성하면 전문성이 자연스럽게 확보됨.
- RAG 문서 소스는 블로그 등 신뢰도 낮은 출처를 피하고, **국내외 진료지침(해마다 업데이트됨) + 최근 몇 년 논문**을 조합하는 방식을 권장. 교수님 본인 프로젝트에서도 동일 방식으로 좋은 결과를 얻었음.
- 고혈압 → 염분, 당뇨 → 당 수치를 사진 한 장으로 자동 분석해주는 흐름이 "만성질환 특화"를 가장 직관적으로 보여주는 지점. 사용자가 직접 타이핑하지 않아도 된다는 편의성과 질환 맞춤 영양 추적을 연결해서 발표에서 강조할 것.

### Q2. AI 피드백(식단·운동·수면)을 하나의 LLM 프롬프트로 통합 처리할지, 각각 따로 할지 어떻게 하는 게 좋을까요?

**교수님 답변**

- 각각 별도 프롬프트로 처리한 뒤 UI에서 합치는 방식 권장. 질문 자체를 "식단만 추천해줘", "운동만 추천해줘"로 분리하면 답변 품질 제어가 쉬움.
- LLM 답변이 장황해지는 문제(예: "그러셨군요~"로 시작하는 긴 서두)는 **프롬프트에 답변 양식을 명시**하는 방식으로 해결 가능. 교수님 본인도 동일한 문제를 겪고 같은 방법으로 해결했음.
- 만성질환 사용자는 중장년층 비중이 높으므로, 답변 길이를 줄이는 동시에 글자 크기와 가독성을 고려한 UI 설계가 함께 필요함.

### Q3. RAG에 과거 기록을 얼마나 포함시키는 게 좋을까요? 만성질환이라 기간 설정이 애매합니다.

**교수님 답변**

- 너무 오래된 기록은 현재 상태에 대한 영향이 적으므로, **최근 5개 기록** 정도로 슬라이딩 윈도우를 고정하는 방식이 현실적.
- 기준은 두 가지를 병행: ① 절대 기준(당 섭취 권장량 초과 여부 등 진료지침 기반), ② 상대 기준(최근 추이 변화 - 예: 당 섭취가 점점 늘고 있음 → 사전 경고).
- 보통은 현재 상태에서 예측을 많이 하지만, 만성질환은 매일 관리하는 특성상 추이 기반 경고도 의미 있음.

### Q4. 수집한 헬스 데이터를 다른 사업과 연결할 수 있을까요? 비즈니스 모델 측면에서요.

**교수님 답변**

- 앱 내 직접 진료는 불가하지만, **"병원 방문 권유" 형태는 가능**. 실제로 이런 구조로 운영되는 병원 주도 헬스케어 서비스 사례가 있음.
- 특정 병원이 만들지 않더라도, 주변 병원을 추천해주는 방식(병원 찾기 앱 형태)도 수익 모델로 검토 가능.
- 데이터를 외부로 보낼 경우 **개인정보 동의 + 비식별화(이름·생년월일 마스킹)** 처리가 선행되어야 하며, 연구 목적으로 활용할 경우 IRB 승인 절차가 필요함(교수님 본인도 승인 과정에서 3차례 거절 경험 있음).

### Q5. 식단 인식 파이프라인을 YOLOv8 → Gemini Vision 2단계로 나눴는데, 실제로 API 호출 비용이 절감되는지 아니면 복잡성만 추가되는 건지 확실하지 않습니다.

**교수님 답변**

- 정답은 **직접 비교 실험**뿐. 단일 Vision 모델 1회 호출 vs 2-stage 호출의 토큰 비용과 정확도를 실측해서 수치로 비교한 결과를 보고서에 포함할 것.
- 실제 상용 AI 서비스에서도 탐지·분류·설명을 각각 다른 모델에 분담시키는 사례가 흔하므로 구조 자체의 정당성은 있음. 다만 "왜 이 구성을 선택했는가"에 대한 비교 근거가 반드시 뒷받침되어야 함.
- 졸프 관점에서 이 비교 분석 자체가 **기여(contribution)** 로 인정받을 수 있음. "해봤더니 됐어요"보다 "후보 A·B의 장단점을 비교했고 이래서 이걸 선택했다"가 훨씬 논리적인 발표가 됨.

## 3. 교수님 추가 코멘트

**발표·심사 관점**

- 어제(6/4) 그로쓰 팀 평가 경험에 따르면, 심사위원들은 기술 구현 방식보다 **"진짜 쓸만한 앱인가", "기존에 없던 차별점이 있는가"** 에 집중하는 경향. 기술 스택은 졸프 팀들 사이에서 거의 동일(RAG 포함 비슷한 구성)하므로 기술 자체로 차별화하기 어려움.
- On-Care의 강점은 **만성질환의 "지속적 관리" 문제를 해결**한다는 점. 귀찮아서 중단하게 되는 것이 만성질환 관리의 최대 장애물인데, 사진 기록의 편의성 + 포인트 제도가 이 지속성을 높이는 데 유효한 수단임.

**포인트 / 게이미피케이션**

- 교수님 본인도 포인트 누적이 앱 사용을 지속시키는 동기가 된다고 언급(현재 약 500회 누적 중). 뉴발런스 앱처럼 포인트 → 실물 혜택으로 연결되는 구조도 참고할 만함.

**트레이너 연계 기능**

- 헬스장 트레이너와 앱 데이터를 연결하는 구조(기록 공유 → 트레이너 피드백, "숙제 검사" 방식)는 기존 앱과 명확히 차별화되는 아이디어로 평가. 혼자 운동할 때 관리 주체가 없다는 문제를 해결하는 접근.

**YOLOv8 한국 음식 인식**

- 현재 오픈소스 YOLOv8 모델은 인식 카테고리가 과일 등 단순 품목 위주로 매우 제한적. 부대찌개 등 한국 음식 인식을 위해서는 **추가 파인튜닝이 필수**. 이 작업 자체가 학술적 기여로 인정받을 수 있음.

**기기 연동**

- 혈당 측정기·혈압계·인바디 연동은 난이도가 높음(교수님 본인 프로젝트에서도 외주 처리). 서버 인프라도 필요하므로 그로쓰 단계 후반 일정으로 배치 권장.

**그로쓰 단계 진행 전략**

- 2학기는 수업과 병행하여 포스터·발표·데모·코드 제출 등 비개발 업무가 집중됨. **방학 중 선행 개발**을 강력 권장. 방학 중 정기 미팅을 유지하며 작업하면 완성도가 크게 높아지는 것을 이전 사례에서 확인함.
