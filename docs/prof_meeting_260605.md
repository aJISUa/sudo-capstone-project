## 2. 면담 질문지

### A. 현재 구현된 결과물 — 기술적 의사결정 검토

**Q1. React 프로토타입 → Flutter 마이그레이션 선택**<br>
Cross-platform 요구사항 측면에서 Flutter 채택이 적절했는지, React Native·Kotlin Multiplatform 등 대안과 비교한 평가.

**Q2. features/* 도메인 모듈 분리 구조**<br>
dashboard / diet / exercise / my_health / ai_coach 등 도메인 기반 모듈 분리가 확장성·테스트성 측면에서 적절한지, 더 나은 패턴이 있는지.

**Q3. Mock 데이터 기반 시연의 학술적 정당성**<br>
현재 LocalApiInterceptor 의 하드코딩 JSON · drift 로컬 DB 로 운영되는 점을 보고서·발표에서 어떻게 솔직하면서도 설득력 있게 기술할지.

### B. 그로쓰 단계 기술 로드맵 — 다음 무엇을, 어떤 순서로

**Q4. MVP 구현 범위의 적절성**<br>
설정한 MVP 기능이 한 학기 결과물로 적절한 수준인지, 그로쓰 단계 진입 전까지 추가해야 할 필수 기능이 있는지.

**Q5. 미구현 / 예정 기능의 착수 우선순위**<br>
Vision AI 파이프라인·RAG 코칭·실 백엔드 등 *그로쓰 단계 예정 기능* 중 가장 먼저 착수해야 할 핵심 모듈.

**Q6. 백엔드 / AI 파이프라인 도입 순서**<br>
그로쓰 단계 시작 시점에 *FastAPI 백엔드 / YOLOv8·Gemini Vision 파이프라인 / RAG 코칭* 중 어떤 순서로 착수해야 학기 내 완성도가 보장되는지.

**Q7. FastAPI 백엔드 우선 설계 컴포넌트**<br>
*(REST API 스키마 / 인증·인가 / 데이터 모델 / 배포 인프라)* 중 가장 먼저 설계·구현해야 학기 내 안정화가 가능한 컴포넌트.

**Q8. 데이터 모델(MySQL RDS) 설계 시 핵심 고려사항**<br>
사용자별 식단·운동·혈압·혈당 시계열 데이터를 다룰 때 *(샤딩 / 인덱싱 / 정규화 vs 비정규화 / 시계열 DB 분리)* 측면에서 우선 고려할 사항.

**Q9. 인증·인가 전략 (소셜 OAuth + JWT)**<br>
현재 mock 인증을 실제 소셜 OAuth (카카오·네이버·Apple·Google) + JWT 로 전환할 때 보안·UX 측면에서 주의할 점.

**Q10. 배포 인프라 선택 (AWS vs GCP vs Vercel 등)**<br>
백엔드 + AI 모델 추론 + DB + CDN 등 전체 스택을 고려할 때 비용·운영 효율 측면에서 권장되는 클라우드 조합.

### C. AI / ML 영역 — 알고리즘·모델·평가

**Q11. Vision AI 2-stage 파이프라인 설계 적절성**<br>
YOLOv8 음식 필터 → Gemini Vision 영양 분석 → 공공데이터 DB 매핑의 2-stage 구조가 적절한지, end-to-end 단일 모델 대비 장단점.

**Q12. YOLOv8 음식 데이터셋·라벨링 전략**<br>
한국 음식 인식 정확도 확보를 위해 *(AI Hub 공공 데이터셋 활용 / 자체 라벨링 / Transfer Learning)* 중 효율적인 접근.

**Q13. Gemini Vision API 의존도와 정확도 검증**<br>
외부 API 에 핵심 기능을 의존하는 구조의 위험과, 결과의 정확도를 어떻게 정량적으로 검증할지.

**Q14. RAG 시스템 평가 메트릭 설계**<br>
*(retrieval recall / answer faithfulness / hallucination rate / domain-grounding)* 등 RAG 평가에서 헬스케어 도메인에 특히 중요한 메트릭과 측정 방법.

**Q15. 헬스케어 LLM 응답의 안전성 가드레일**<br>
"의료 행위"로 오해될 수 있는 LLM 답변을 차단하는 프롬프트·시스템 메시지·후처리 가드레일 설계 가이드.

**Q16. 개인화 모델의 cold-start 문제**<br>
신규 사용자의 누적 데이터가 부족할 때 RAG·추천 알고리즘이 어떻게 의미 있는 결과를 내야 할지에 대한 기법.

### D. 코드 품질 / 엔지니어링 관행

**Q17. 코드 품질 현재 수준 평가**<br>
디자인 시스템·테스트·l10n·라우팅 등 기반 작업의 학술적 평가, 우선 보완해야 할 영역.

**Q18. 테스트 전략 — 현재 커버리지의 적절성**<br>
현재 단위 / 위젯 / golden 테스트 위주 구성이 적절한지, 추가해야 할 영역 *(통합 테스트 / E2E / 시각 회귀 / 성능 테스트)*.

**Q19. CI/CD 파이프라인 강화 방향**<br>
현재 GitHub Pages 배포 워크플로우 외에 추가해야 할 자동화 *(린트·테스트·빌드 검증 / 보안 스캔 / PR 자동 라벨링)*.

**Q20. 코드 리뷰 관행과 PR 컨벤션**<br>
3인 팀에서 효과적인 코드 리뷰 운영 방식, PR 단위·크기·머지 정책의 권장 패턴.

### E. 데이터 / 보안 / 규제

**Q21. 실사용자 테스트 및 데이터 수집 계획**<br>
그로쓰 단계에서 실사용자 피드백·로그 수집 과정을 어느 시점부터 포함할지, 기능적 고도화와의 우선순위.

**Q22. 개인정보 보호 및 IRB 절차**<br>
실사용자 데이터를 수집할 경우 *식단 사진·혈압·혈당* 같은 민감정보의 법적·윤리적 보호 장치, IRB / 학내 윤리위원회 절차 필요 여부.

**Q23. 의료기기·디지털 치료기기 규제 검토 시점**<br>
On-Care 가 *고혈압·당뇨 위험군 관리* 를 표방하는 만큼, 의료기기·디지털 치료기기 분류 가능성과 검토가 필요한 시점.
